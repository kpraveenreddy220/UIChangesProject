Startup Overview

When the Luxe system boots, a system daemon named tm-core scans the various sub-directories under the /home directory to create a list of applications.

Each application may define the resources and how it should be started up through a file called app.conf that should be in the root of the application's directory. For example, the application viking should have a control file in 

 /home/viking/app.conf

The syntax for this format file follows 

<?xml version="1.0" encoding="UTF-8"?>
This is a first line of the file.

 

<application>
This is the outer enclosing tag.

 

<home_button form="desktop/index.html"/>
There can only be ONE app.conf that specifies this tag. This is the HTML page that will be shown when the home button ( magnifying glass + 0 + 0 ) is pressed. The attribute "form" specifies the path that is passed to the page show function.

For example, page/show?path=desktop/index.html will show the page /home/desktop/html/index.html on the display.

 

<autolaunch path="/home/sdkdemo/appfcgi -d" default_form="html/index.html"/>
autolaunch path= If an application needs to run an executable on a boot up, the autolaunch path= attribute specifies the location of the binary and any parameters required by the binary.

This may be used to run daemons such as an application that requires a FastCGI interface to the web server.

autolaunch form= There must only be ONE app.conf that specifies an autolaunch form= attribute. This specifies the HTML page that will be shown when the terminal boots.

The attribute "form" specifies the path that is passed to the page show function.

For example, form="desktop/index.html" will call page/show?path=desktop/index.html to show the page /home/desktop/html/index.html on the display.

 

<desktop_item form="app/index.html" caption="Point of Sale" icon="files/pos.png" icon_position="top left"/>
More than one application may include a tag. If an application would like to appear on the Desktop, it should include a tag in its app.conf file.

The desktop manager will display the icon specified in the "icon" attribute along with the brief description specified by the "caption" attribute.

The Desktop manager will enable a touch input area that will show the page specified by the attribute "form" which specifies the path that is passed to the page show function when the desktop icon selected .

For example, page/show?path=setup/index.html will show the page /home/setup/html/index.html on the display.

 

<setup_item form="pinpad/setup.html"/>
More than one application may include a tag. If an application can be configured by a common setup application, it should put the path of its setup page in the file.

 

<resources>resources/resources.xml</resources>
If an application has secure resources to load into MQX on a boot up, this tag define the location of a file that specifies the list of resources to load.

 

<prompts>resources/prompts.json</prompts>
Although we could simply state that applications that want to specify application specific prompts must put them into their applications resources directory, such as /home/APP_NAME/resources/prompts.json, as we have a tag for the secure resources , to be consistent and to allow for prompts to be in either the resources/ or files/ directory, the tag is used to specify the location of the prompts file.

 

<card_ranges>resources/card_ranges.xml</card_ranges>
<white_list signature=... >whitelist.xml</white_list>
These xml tags are used by the payment engine to determine the encryption schemes that are applied to a card PAN.

 

Other EMV Configuration XML file(s) 

<app_specific_tags></app_specific_tags>
Each application may have their own specific tags that only it requires.
